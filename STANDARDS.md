# Standards тАФ the global, env-aware engineering rules

The portable rules that apply to **every** project I work on (ComfyUI, Astro, React Native, тАж),
loaded by the workflow skills (`/fire`, `/polish`). App-specific conventions (a project's styling
system, state layer, i18n) do NOT live here тАФ they stay in that app's `CLAUDE.md`. **Detect the
project's stack first** (read `package.json` / `pyproject.toml` / framework config) and apply only
the sections that match.

## Universal (every project, every language)

- **Never guess тАФ check the source first.** Verify a command / API / library / error against the
  actual source (the repo's files, its OPEN **and** CLOSED issues + PRs, current docs) before acting.
  On a bug, the fix is often already upstream. **No hacked workarounds** тАФ find the proper, current
  best-practice method; a hack is the last resort and only with explicit approval.
- **Prove, don't theorize.** Run the script, read the file, reproduce the failure and then the
  success. Empirical beats plausible.
- **LSP over grep for symbols.** Use language-server `goToDefinition` / `findReferences` / `hover` /
  `documentSymbol` for anything about symbols (prove "zero references тЖТ safe to delete" before
  deleting). Grep is only for raw text patterns. **If the LSP server is down, STOP and say so** тАФ
  never silently fall back to grep for symbol work.
- **Don't write to repos we don't own.** Research third-party/upstream repos read-only; never
  comment, file issues/PRs, or push to them. A finding that belongs upstream тЖТ tell the user, stop.
- **Minimal, focused changes.** Don't refactor unrelated code, don't add unspec'd "nice to haves",
  preserve existing patterns.
- **Plans live in-repo at `docs/plans/<id>.md`.** Anything bigger than a one-liner gets a committed
  plan file keyed to its roadmap/task id тАФ the durable spec a fresh-context agent reads (survives
  `/clear`, lives in git, travels with the repo). The roadmap's `> nerd:` line becomes a thin pointer
  to it; don't inline the plan into the roadmap. The harness plan-mode file under `~/.claude/plans/`
  is machine-local scratch тАФ copy the approved plan into `docs/plans/<id>.md`. `/fire` reads this file
  first (if present) and treats it as authoritative over the nerd line.
- **Commits:** conventional messages (`feat:`/`fix:`/`chore:`тАж), one concern each. **Local branch
  only тАФ never push, never the default branch (`main`/`master`) automatically.** **Never add AI
  attribution** тАФ no `Co-Authored-By`, no "generated by" trailer.
- **Git hygiene:** never track secrets or per-machine config. Ignore all env files but a template
  (`.env` / `.env.*` / `!.env.example`); split Claude config into committed `settings.json` (shared)
  + gitignored `settings.local.json` (per-machine).
- **Close by naming the next rung тАФ never hand-wave "push the branch?".** End every workflow skill's
  Report with the *exact* next skill to run, decided by git state (`git status --porcelain` +
  `git rev-list --count <trunk>..HEAD`): **uncommitted changes** тЖТ the commit skill (`/fire` app ┬╖
  `/forge` skill) first, *then* `/land`; **committed, app work** тЖТ `/ship` to archive, then `/land`
  to publish; **committed, already archived / skill work** тЖТ `/land`. `/land` is the only rung that
  pushes тАФ name it explicitly; don't ask a vague "want me to push?".
- **Parallel work = one git worktree each тАФ never two builds in one checkout.** A repo has a single
  HEAD per working tree, so concurrent `checkout`/commit from two agents corrupts *which branch a
  commit lands on* (the mismatch). To build more than one id at once, spawn each in its own worktree
  (the Agent tool's `isolation: "worktree"`) тАФ one id тЖТ one branch per worktree, fully isolated.
  No worktrees? **Serialize** тАФ one build at a time in the shared checkout. Land serially regardless:
  `/land` the first (ff-only), rebase the next onto the new tip, land it; `/land` refuses a non-ff so
  a tangled merge can't happen by accident. Single-threaded work needs no worktree.

## Verification gate (auto-detect per project)

- **"Verify" = the project's own static gate, run once at the end, clean.** Detect it from the
  project, don't hardcode:
  - `package.json` scripts тЖТ `typecheck`/`tsc --noEmit`, `lint`, `test` (run those that exist).
  - Python (`pyproject.toml`/`uv`) тЖТ `ruff`/`mypy`/`pytest` as configured (the user runs Python via
    **uv**, never bare `python`).
  - Else тЖТ the framework's build/check equivalent.
- **Invoke the toolchain directly** тАФ `npx` / `npm` / `uv` etc., assuming node/tooling is on PATH.
  Don't prepend a version-manager activation dance (no `fnm env`/`nvm use` ceremony); if a binary
  truly isn't found, locate and call it, don't bake an activation step into the project's docs.
- **Device / emulator / runtime / visual checks are the USER's job**, not the gate. Finish the
  static gate тЖТ the automated part is done; the user runs it and reports findings.

## TypeScript / JavaScript projects

- **Strict mode on. Never use `any`** тАФ no `as any`, no new `eslint-disable no-explicit-any`. For a
  genuinely unknowable value use **`unknown` and narrow it** before use.
- Type all component props, public functions, and data models (`interface`/`type`, no `I` prefix).
- **One-line doc comment (`/** тАж */`) on every exported symbol, written WITH the code, not as a later
  pass.** Document *why*, not *what*. New exported code isn't "done" without it.
- No commented-out code, no unused imports/variables. Type-check + lint clean before "done".
- Naming: PascalCase components/types, camelCase functions/vars, SCREAMING_SNAKE_CASE consts.

## React Native / Expo projects (apply only if the stack matches)

- **Target the project's exact SDK** тАФ check the versioned Expo docs before using a module; don't
  assume older API shapes. Reach for libs already in the project before adding deps.
- Write **Android-safe by default** (iOS is lenient, Android strict) and use **`/rn-debug`** when a
  screen renders on one platform and not the other тАФ its **Known Issues** registry is the portable
  RN/Expo mega-bug catalogue (inputs/caret, keyboard scroll, sheets, NativeWind, a11y, native modules).
- Functional components + hooks; one job per component; extract reusable logic to hooks and
  app-agnostic UI to a `packages/`-style location.
- Test UI changes in **both** RTL and LTR if the app is bilingual; use logical start/end, not
  hardcoded left/right.
