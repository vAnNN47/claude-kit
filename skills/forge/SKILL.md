# forge — execute one SKILLS-TODO item

Take a single `[slug]` from `SKILLS-TODO.md` to a built, load-checked, archived skill change.
The **twin of `/fire`** — same spine, but its queue is the claude-kit tooling backlog instead of
an app's `roadmap.md`, and its gates are a **load-check + self-test** (`/skill-tester`) instead of
tsc+lint — a skill must both *load* and *behave*.

> **Why this exists:** `inbox` *fills* `SKILLS-TODO.md` (it routes tooling tasks there), `fire`
> drains `roadmap.md`, and `ship` archives finished app work — but nothing **drained the skills
> queue**. This is that missing rung.

All work happens **in the claude-kit repo** (`~/.claude/skills/claude-kit/`, its own git repo on
`master`) — not in the app you're using it from.

## Steps

1. **Locate.** Read `SKILLS-TODO.md`; match `<slug>` to an open `- [slug]` line under
   `## Ideas / gaps`. Empty arg → list the open items and ask which to forge. No match → list the
   closest and stop; don't guess which item the user meant. Items under `## Capability gaps` are
   **flagged-only** — refuse to auto-build one. Tell the user to promote it to an idea first (this
   mirrors `inbox`'s "flag, never build" rule).

2. **Restate, then classify.** First action, before any edit: restate the item in one sentence —
   "this item asks me to ___" — to pin scope. Then pick *how* you build:

   | Kind of item | How to build it |
   |---|---|
   | **New skill** | Author `skills/<name>/SKILL.md` in house style. For a skill with **objectively verifiable output**, offer to run the official **`skill-creator`** draft→eval→iterate loop (the rigorous engine). For workflow-glue / subjective skills, author directly — say so and skip the eval machinery. |
   | **Tweak** (docs, framing, an existing `SKILL.md` / `README.md`) | Edit the relevant file directly. |
   | **Marketplace adopt** | Discover with `npx skills find <query>`, add with `npx skills add <owner/repo> --skill <name>`. Record the **security scan** in the archive line. |

3. **Branch.** In the claude-kit repo, cut a fresh branch off `master`: `chore/forge_<slug>`
   (tooling work = `chore`, matching fire's prefix scheme). Never commit on `master`.

4. **Build it for real**, following claude-kit frontmatter rules (`name` == folder; `description`
   and `argument-hint` are strings; a bracketed `argument-hint` is quoted) and the terse
   inbox/fire/ship tone. Add a **"Not this skill's job"** footer to any new skill. For a new
   skill, also author its **`TEST.md`** spec (trigger, sandbox `setup:`, `expect:` side effects,
   `guardrails:`) next to the `SKILL.md` — the next step runs it. A skill that genuinely can't be
   sandboxed (it mutates a real repo) gets `mode: load-check-only`.

5. **Load-check (first gate).** Re-read the frontmatter of every `SKILL.md` you touched: it must
   load — `name` equals its folder, `description`/`argument-hint` are strings, bracketed
   `argument-hint` quoted. (No tsc — these are docs.) Note that the user should **reload the VS
   Code window** to see a newly added skill.

6. **Self-test (second gate).** Run **`/skill-tester <name>`** on the skill you just built. It
   sandboxes the skill, drives it via its `TEST.md`, and asserts the real side effects +
   guardrails. **PASS → continue. FAIL → stop**, leave the work on the branch, and report which
   assertion broke — do not commit or archive a skill that fails its own test (the same spirit as
   fire's "do not commit broken code"). A `load-check-only` spec passes this gate at step 5.

7. **Commit.** One conventional commit in the claude-kit repo; subject from the slug + plain line
   (e.g. `feat(forge): add link-preview skill [notes_link_preview]`). **No co-author / attribution
   footer** — do not add `Co-Authored-By` or any "generated by" line. Local only — never push.

8. **Close the loop.** Move the item's `- [slug]` line **out of `SKILLS-TODO.md`** into
   `archive/[year]/[mm-month].md` (e.g. `archive/2026/06-june.md`; create the file/dirs if missing)
   under today's `## YYYY-MM-DD` heading, as:
   ```
   - [slug] (skill) — what shipped → branch · commit
   ```
   `SKILLS-TODO.md` stays **OPEN-only** — never grow a `## Done` section; that archive line is the
   whole record. If a new skill landed, also add its row to the **skills table in `README.md`**. An
   item lives in exactly one place — open, then archived.

9. **Report.** What you built, the load-check + self-test results, the branch + commit, and
   anything left open.

## Rules

- **Exactly one slug per call.** Batching = run forge again. Keeps branches/commits isolated and
  revertible.
- **Local only.** Never `git push`, open a PR, or merge. Remote git is the user's job.
- Load-check is mandatory — shipping a skill that won't load is the failure mode this gate prevents.
- Self-test (`/skill-tester`) is mandatory for a new/changed skill — a skill that loads but mangles
  its side effects (wrong format, stray `Co-Authored-By`, a `git push`) fails here, not in prod.
- **Move, never delete** on archive — the line lives in the archive afterward, not nowhere.

## Not this skill's job

- Sorting a brain-dump into the queue → `/inbox`.
- The rigorous draft→eval→optimize **engine** for one skill → `skill-creator` (forge *delegates* to
  it for objectively-testable new skills; it doesn't replace it).
- App feature/bug/tech-debt work on an app's `roadmap.md` → `/fire`.
- Archiving finished *app* work → `/ship`.
